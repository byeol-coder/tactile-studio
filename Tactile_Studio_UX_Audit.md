# Tactile Studio — UX·접근성 감사 및 개선안

대상: `byeol-coder/tactile_agent` (`main`, dc-runtime 프로토타입)
관점: 시니어 프로덕트 디자인 · UX 아키텍처 · 접근성 · 촉각 그래픽 제작

---

## 0. 먼저 — 현재 상태에 대한 솔직한 진단

브리프는 "엔지니어링 데모, 기능이 한 번에 다 노출됨"을 전제로 하지만, **실제 `main` 빌드는 그 전제를 이미 상당 부분 넘어서 있습니다.** 코드를 읽어보면:

- 허브 → 4개 생성 진입점(명령어·이미지·템플릿·빈 그리드)이 이미 구현됨. `entCmd`(추천 배지) + `startImage`가 1차 위계로 강조되어 있음 — 브리프의 "Create from Prompt / Image 우선" 방향과 일치.
- 고급 변환 설정(otsu/adaptive/alpha/contour/denoise 등)은 **첫 화면에 노출돼 있지 않음.** 사전(`t`)에 해당 라벨이 아예 없음 → 이미 인스펙터/리뷰 단계로 이동 완료.
- `Send to DotPad`가 primary CTA로 라벨링됨(`send: 'Send to DotPad'`). DotPad 상태머신(idle/connecting/sending/done/error/incompat)이 `sendPhase`로 완비.
- 하단 상태바, 우측 contextual inspector, 5단계 이미지 변환 위저드(가져오기→자르기→변환→검토→편집), before/after 미리보기, 품질 라벨(`qGood/qSparse/qDense`) 모두 존재.
- 접근성 기반이 탄탄함: 버튼 105개 중 aria-label 84개, `:focus-visible` 대응 스타일 87곳, `role="dialog"`+`aria-modal` 8쌍, `role="status"` 13곳, `role="alert"` 3곳, `aria-live` 라이브 영역, KO/EN 완전 이중화.

**결론:** 이건 "재설계가 필요한 데모"가 아니라 "완성도 높은 프로토타입의 마감·정합성 문제"입니다. 그래서 이 문서는 처음부터 다시 그리는 대신, (1) 실측으로 잡아낸 **진짜 결함**과 (2) 10x 체감을 만들 **정합성·마감 항목**에 집중합니다.

## 1. UX 감사 요약

강점(유지):
- 진입 위계, 컨텍스트 인스펙터, DotPad 상태머신, 이중언어, 키보드/SR 기반이 이미 프로덕션급.

실제 갭(우선순위 순):
1. **대비(Contrast) — 유일한 명백한 AA 결함.** 보조 텍스트 `#9C9994`가 밝은 배경에서 2.6–2.8:1(4.5:1 미달). 브랜드 주황 `#EC5927`은 흰 배경 텍스트/버튼 라벨 기준 3.49:1(일반 텍스트 4.5:1 미달). → §4.
2. **노출된 기술 용어 잔존.** `임계값 / Threshold` 1개가 사용자 대면. 나머지(denoise/contour/binarize)는 이미 숨겨져 있어 이 하나만 남음. → §5. (적용 완료)
3. **위계의 미세한 흔들림.** 허브 CTA에서 이미지/명령어가 1차라는 점이 시각적으로 더 강조될 여지(버튼 크기·색 대비). 브리프 의도 대비 "동등해 보이는" 카드가 일부 있음.
4. **첫 30초 답의 명시성.** "이게 어떻게 촉각 그래픽이 되는가"(3번 질문)는 위저드 안에선 훌륭하나, 허브에서 한 줄로 미리 보이면 전환에 유리.

## 2. 정보 구조(IA) 제안

현재 구조를 유지하되 **위계 라벨만 명확화**합니다(구조 변경 최소화가 곧 안전):

```
Tactile World Hub
├─ [1차] 이미지로 만들기      ← 가장 크고 강한 카드
├─ [1차] 명령어로 만들기(추천)  ← 동급 강조 + '추천' 배지 유지
├─ [2차] 템플릿에서 시작       ← 보조 카드
└─ [2차] 빈 그리드            ← 보조 카드
        │
        ▼
Tactile Studio (편집기)
├─ 상단바: 로고 · 파일명 · 저장 · 내보내기 · [Send to DotPad ★primary]
├─ 좌측 레일: 선택 · 펜 · 지우개 · 도형 · 묵자 · 빠른작업  (아이콘+라벨/툴팁)
├─ 중앙: 촉각 캔버스 (시각적 주인공)
├─ 우측 인스펙터: 컨텍스트별 (변환설정 / 브러시 / 묵자 / 선택객체)
└─ 하단 상태바: 점 수 · 좌표 · 가독성 · 밀도 · 구조 명료도 · DotPad 상태 · 마지막 전송
```

권고: 2차 카드(템플릿·빈 그리드)는 1차 카드보다 **면적·대비를 한 단계 낮춰** "동등해 보임"을 제거. 현재 split 뷰의 두 카드가 시각 무게가 비슷하므로 1차 쌍만 채움색/큰 사이즈로 승격.

## 3. 워크스페이스·레이아웃 권고 (마감 항목)

- **캔버스 우선순위:** 인스펙터·상태바의 시각 노이즈를 낮춰 캔버스가 확실한 hero가 되도록. 상태바는 이미 compact 하나, 6개 지표를 "핵심 3(점수·밀도·DotPad) + 접힘 3"으로 그룹화하면 스캔성 향상.
- **레일 라벨:** 아이콘-only 버튼은 aria-label 존재(양호). 마우스 사용자를 위해 hover 툴팁 텍스트가 라벨과 동일 문구인지 1:1 점검(중복 안내 방지).
- **DotPad 고급 컨트롤(live/pin/invert/refresh):** 이미 패널 하위에 그룹화됨 → "고급 DotPad 컨트롤" 접이식 라벨만 명시화 권고.
- **Send 성공 순간:** `sentSub`("DotPad에서 출력되고 있습니다")가 이미 좋음. 성공 토스트에 "촉각 미리보기 확인" CTA 한 개만 유지해 과잉 방지.

## 4. 접근성 AA 이슈 목록 + 수정값 (실측)

WCAG 2.1 상대휘도 공식으로 실제 팔레트를 계산한 결과:

| 항목 | 조합 | 실측 | 기준 | 판정 |
|---|---|---|---|---|
| 보조 텍스트 `#9C9994` / 흰 배경 | text | **2.84:1** | 4.5 | ❌ |
| 보조 텍스트 `#9C9994` / 웜 배경 `#F7F4EF` | text | **2.59:1** | 4.5 | ❌ |
| 브랜드 주황 `#EC5927` 텍스트 / 흰 배경 | text | **3.49:1** | 4.5 | ❌(일반)·✅(large/UI) |
| 흰 글자 / `#EC5927` 버튼 | 14–15px 라벨 | **3.49:1** | 4.5 | ❌(비-large) |
| 카드/구분선 `#EAE1D7` / 흰 | 장식 | 1.29:1 | — | 면제(1.4.11) |
| 입력·컨트롤 경계 `#EAE1D7` | UI | 1.29:1 | 3.0 | ❌(기능 경계) |
| `#6B6862` / 흰·웜 | text | 5.55 / 5.06 | 4.5 | ✅ |
| `#C43D00` / 흰 | text | 5.23 | 4.5 | ✅ |
| 본문 `#1E1C1A`, 초록 `#0B8800`, 빨강 `#DA120D` | text | 15.5 / 4.63 / 5.16 | 4.5 | ✅ |

**수정 토큰(별님 Figma AA 결정과 일치):**
- 밝은 배경의 `#9C9994` → **`#6B6862`** (기존 팔레트 값, 5.0+:1). ⚠️ **어두운 배경 위 `#9C9994`는 유지**(대비 반대 방향). 유지 대상 라인: `87, 89, 221, 222` (히어로 eyebrow/서브, 임베드 브레드크럼/배지).
- 주황을 **의미 분리**: `brand-orange #EC5927`(어두운 배경·큰 장식·10px 상단바 유지) vs `orange-strong #C43D00`(밝은 배경의 주황 텍스트·흰글자 CTA 채움색). 어두운 히어로 CTA는 `#EC5927` 유지(어두운 배경에선 더 밝은 쪽이 가독성↑).
- 기능적 입력 경계는 `#EAE1D7` 대신 `#C9BBAF`(3:1 근접) 이상으로. 순수 장식 구분선은 그대로 두어도 됨.

**이미 적용 완료(코드):** 눈으로 밝은 배경임을 확정한 8곳(`#9C9994`→`#6B6862`): 메인 서브타이틀, 상단 내비, 아바타, split-허브 카드 서브, 돌아가기 버튼, 출력 라벨 아이콘, 스테퍼 todo 숫자·라벨.

**남은 `#9C9994` 36곳** — 부모 배경 상속 때문에 렌더링 없이 100% 확신 불가. 아래 §7의 스니펫으로 일괄 적용 후 **시각 확인 1회**가 안전. 손으로 확인 권장(어두울 가능성): `670, 684, 685, 711, 935, 974, 1135` (DotPad 미리보기/기기 프레임 계열).

키보드·SR(이미 양호, 유지 점검만):
- 모달 focus trap + ESC 닫기 + 트리거 복귀: `role=dialog/aria-modal` 존재 → 트랩·복귀 로직 실동작 회귀 테스트만 1회.
- 캔버스 연속 조작 중 aria-live 반복 announce 억제: `aDotOn/aDotOff`가 매 셀마다 울리지 않도록 디바운스(이미 `hideTip` 패턴 있음 — 동일 적용).
- 파일 업로드: 보이는 "파일 선택" 버튼 존재(`impChooseFile`) + 드롭 → 키보드 대안 확보됨. 유지.
- reduce-motion: `@keyframes` 다수 사용 → `@media (prefers-reduced-motion: reduce){ *{animation:none!important;transition:none!important} }` 1블록 추가 권고(§7).

## 5. UX 라이팅

브리프의 라이팅 권고는 **대부분 이미 반영됨**(Send to DotPad, Convert image, 빈 그리드, 플레인 문구). 실제로 사용자 대면에 남아 있던 기술 용어는 `임계값/Threshold` 하나뿐:

| 이전 | 변경(적용됨) |
|---|---|
| 임계값 / Threshold | 촉각 디테일 정도 / Tactile detail level |
| "임계값을 조정해…" | "촉각 디테일 정도를 조정해…" |
| "Adjust the threshold…" | "Adjust the tactile detail level…" |

`디더링/Dithering`은 의도적으로 유지 — denoise가 아니라 별개의 정당한 컨트롤이고 리뷰 단계 내부에 있어 조기 노출이 아님. (denoise/contour/binarize는 현재 빌드에 미노출이라 손댈 것 없음.)

## 7. 적용된 코드 변경 + 다음 패치 스니펫

**적용 완료 파일:** `index.patched.html` (원본 대비 정확히 12곳 변경, `<x-dc>`/`dc-script` 구조 무결 확인).

**남은 대비 일괄 적용(로컬에서 실행 → 렌더링 시각 확인):**
```python
# 어두운 배경 위 #9C9994는 제외하고 밝은 배경만 #6B6862로.
DARK = {87,89,221,222,670,684,685,711,935,974,1135}  # 유지(어두운 배경) — 시각 확인 후 조정
lines = open('index.html', encoding='utf-8').read().split('\n')
for i,_ in enumerate(lines):
    if (i+1) not in DARK and '#9C9994' in lines[i]:
        lines[i] = lines[i].replace('#9C9994', '#6B6862')
open('index.html','w',encoding='utf-8').write('\n'.join(lines))
# 적용 후 히어로/임베드 바/DotPad 프레임을 눈으로 확인 — 어두운 곳에 #6B6862가 남았다면 되돌리기.
```

**reduce-motion(허브 `<style>` 블록에 1줄 추가):**
```css
@media (prefers-reduced-motion: reduce){*{animation-duration:.001ms!important;animation-iteration-count:1!important;transition-duration:.001ms!important}}
```

**주황 토큰 분리(권고, 미적용):** 밝은 배경의 주황 텍스트·흰글자 CTA 채움색만 `#EC5927`→`#C43D00`. 어두운 히어로 CTA·상단 10px 바는 `#EC5927` 유지.
